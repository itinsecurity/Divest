## Executive Summary

### Amendments Required
1. **[CRITICAL] Instrument Search/Disambiguation UX** — New FR required for search/select disambiguation flow in holding registration. FR-005 amended to ISIN-first matching with ticker fallback requiring user confirmation. New acceptance scenario for multi-match and zero-match cases.
2. **[HIGH] AI Agent Secondary Enrichment Split** — FR-007 split into FR-007a (primary API enrichment, P1) and FR-007b (AI agent, P2/P3 with spike prerequisite). SC-002 scoped to FR-007a only. Manual editing (User Story 4) promoted from P3 to P2 as interim fallback.
3. **[HIGH] Ticker-Based Matching Ambiguity** — FR-005/FR-008 amended to mandate ISIN as primary match key; ticker fallback only when ISIN unavailable and requires user confirmation. Resolved jointly with #1.
4. **[HIGH] Delete/Edit Holdings Missing User Stories** — New P1 user story required for FR-024/FR-025 with acceptance scenarios covering delete, edit (shares/value/account), shared profile survival, and orphan profile handling.
5. **[HIGH] Stale Portfolio Data** — New FR for "last updated" timestamp on holdings. FR-025 explicitly scoped to include price/value editing. New acceptance scenario for price update workflow.
6. **[HIGH] Combination Fund Without Split Data** — FR-020 amended: combination funds without equity/bond split data excluded from stock/interest calculation and shown in "Unclassified" bucket. New acceptance scenario in User Story 3.
7. **[MEDIUM] Spread Views Data Coverage Indicator** — New FR: spread views MUST display percentage of total portfolio value represented in the analysis and count of excluded holdings. Strengthens FR-021's vague "clearly indicated" language.
8. **[MEDIUM] Shared Profile Edit Confirmation** — New FR requiring confirmation dialog when editing a shared profile, disclosing how many holdings are affected.
9. **[MEDIUM] Enrichment SLA Scoping** — SC-002's 30-second SLA applies only to FR-007a (primary API enrichment). AI agent path (FR-007b) to receive separate SLA after spike.

### Decisions Made During Debate
- **Orphan profile policy**: When the last holding referencing an asset profile is deleted, the profile is **retained** (not garbage-collected). Rationale: re-enrichment is expensive and profiles are small. (Raised by Scrum Master, resolved by Product Owner.)

### Resolved
- **SC-003 Norwegian coverage** (Business Sponsor, MEDIUM) — Accepted as-is; the 3-of-4 threshold plus API selection during planning provides adequate guard rails.

## Debate Highlights

### Concerns #1 + #5: AI Agent Enrichment and SLA (Three-way consensus: TL, BS, SM)
The three roles most concerned with delivery risk aligned quickly. Scrum Master proposed the P1/P2 split; Tech Lead refined it into FR-007a (primary API, P1) and FR-007b (AI agent, P2/P3 with spike prerequisite). Business Sponsor added a key insight: promoting manual editing (User Story 4) from P3 to P2 creates a complete data-completeness toolkit at the point users first see spread views. This also resolved Concern #5 — the 30-second SLA becomes realistic when scoped to predictable API calls only. No disagreement from any participant.

### Concerns #2 + #3: Ticker Ambiguity and Instrument Disambiguation (Three-way consensus: TL, UX, PO)
Tech Lead and UX Designer recognized these as two sides of the same problem. UX Designer proposed a search/select disambiguation flow; Tech Lead linked it to backend ISIN-first matching. Product Owner confirmed that if users select instruments through a disambiguation UI that surfaces ISINs, ticker collisions are mitigated at the point of entry. UX Designer upgraded Concern #3 to CRITICAL with PO and TL support — this is foundational to User Story 1 (P1). The joint resolution produces both a UX amendment (new FR for disambiguation) and a data integrity amendment (FR-005 ISIN-first matching).

### Concern #4: Delete/Edit Holdings (Rapid consensus: PO, SM, BS)
Fastest consensus of the meeting. Three roles independently agreed these are table-stakes CRUD operations that need P1 user stories with acceptance scenarios. Scrum Master added a refinement: the new story must specify orphan profile handling (what happens when the last holding referencing a profile is deleted) as this is a product decision that affects estimation. Product Owner acknowledged this needs clarification.

### Concern #6: Stale Portfolio Data (Three-way consensus: BS, PO, UX)
Business Sponsor originally raised this as MEDIUM. UX Designer's argument that "the portfolio becomes fiction without the user knowing" convinced Product Owner to upgrade to HIGH. Three concrete amendments agreed: (1) new FR for "last updated" timestamp, (2) FR-025 explicitly includes price/value editing, (3) new acceptance scenario for price update workflow.

### Concern #7: Combination Fund Without Split Data (Consensus: PO, TL, UX)
Tech Lead framed this as a "confidently wrong" problem — the system would silently attribute a combination fund's full value to one category rather than showing uncertainty. Product Owner upgraded to HIGH. The "Unclassified" bucket solution emerged: combination funds without split data are excluded from stock/interest calculation and shown separately. UX Designer extended this principle to sector/geographic views — never silently drop or misattribute data.

### Concerns #8 and #9: Shared Profile Edit + Data Coverage (No challenges)
UX Designer's Concern #8 (shared profile edit confirmation) received no challenges — stands as MEDIUM with a new FR needed. Concern #9 (data coverage indicator) was actively supported by Business Sponsor, who confirmed it's a trust and decision-quality issue, not just a UX nicety. Both amended with concrete FR language.

## Concern Resolutions

### C1: AI Agent Secondary Enrichment as P1
**Severity:** HIGH
**Raised by:** Business Sponsor, Tech Lead, Scrum Master
**Spec Item:** FR-007

**Debate Summary:**
Three-way consensus reached quickly. Scrum Master proposed the split; Tech Lead provided the FR-007a/007b structure; Business Sponsor added the manual editing promotion to P2. No disagreement.

**Resolution:** AMENDED
- Split FR-007 into FR-007a (primary API enrichment, stays P1) and FR-007b (AI agent secondary enrichment, moves to P2/P3)
- FR-007b requires a feasibility spike before commitment
- Promote User Story 4 (manual editing) from P3 to P2 as interim fallback
- SC-002 (30-second SLA) scoped to FR-007a only
- Agreed priority regrouping:
  - **P1**: Register holdings + primary API enrichment + delete/edit holdings
  - **P2**: Spread views + manual profile editing + AI agent enrichment (spike first)
  - **P3**: Profile refresh
- Sprint sequence: Sprint 1 (API enrichment + holdings CRUD) → Sprint 2 (manual editing + spread views + AI spike) → Sprint 3 (AI agent if spike succeeds)

**Raised by:** Tech Lead
**Spec Item:** FR-005/FR-008
